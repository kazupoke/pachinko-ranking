# マイパチ店 ゲーム設計

実店舗経済ベース・200 台規模のホールを運営するシミュレーション。

## 1. 全体ループ

```
(放置収入) ← 稼働率に応じて資金じわ増 ★
      ↓
(ガチャ) 新台抽選 (1日1無料 + 3日で10連分チケット)
      ↓
(新台配置) MyShop に設置 → 稼働率・単価↑ ★
      ↓
(客層・設備) 集客が変化 → 故障発生
      ↓
(修理 / 設備投資) 資金消費 → 集客維持
      ↓
(覆面調査) 他店訪問で追加報酬 ★
```

★ = 既に実装済み

## 2. 既存実装の把握

現行コード ([useGameStore.ts](game/src/stores/useGameStore.ts)) に以下が揃っています:

| 機能 | 実装箇所 | 内容 |
|---|---|---|
| 初期資金 | `STARTING_CASH` | 500 万 G スタート |
| ログインボーナス | `claimLoginBonus` | 50 万 G × streak 最大 7 日 |
| 稼働シミュレーション | `tickSimulation` | 新規客 × 単価5,000 × 粗利20% / オフライン12h まで |
| ガチャ天井 | `gachaPity` | - |
| 覆面調査 | `submitMysteryReport` | 1 日 3 件 / FML 重みで報酬 |
| CPU 店舗 | [seed_shops.ts](game/src/data/seed_shops.ts) | プリセット複数店舗 (tier 1-5) |
| 店舗拡張 | `expandMachineSlot` / `expandTypeSlot` | 200→400 台、40→60 機種 |
| 台の設置 / 撤去 | `installMachine` / `uninstallMachine` | 上限内で自由配置 |

## 5. ガチャ仕様

- 無料: **毎日 1 回単発 + 3 日で 10 連チケット 1 枚蓄積**
- 排出: N 60% / R 30% / SR 8% / SSR 2%
- 天井: 100 回で SSR 確定 (既存 `gachaPity` を利用)

## 6. 未実装機能とロードマップ

### M1: スロット実遊戯 (Phase 3)

演出版を採用:
- 設置済み台を選択 → リール回転 3〜5 秒 → 停止 → 当選判定
- 1 プレイ ±100 〜 ±10,000 G
- 機種ごとに設定値 (1〜6)
- 10 回プレイでクールダウン

### M2: 台の耐久度・故障・修理 (Phase 4)

- 台ごとの `durability` (0-100)
- `tickSimulation` 内で稼働量に応じて減少
- 客層悪化で減少速度 2〜3 倍
- 0 で稼働停止 → 修理費 50,000 G
- `shop.layout[i].durability` として持たせる

### M3: 設備システム (Phase 4b)

| 設備 | 故障時の影響 | 修理費 |
|---|---|---|
| トイレ | 集客 -15% | 30,000 G |
| 空調 | 夏場の稼働率 -20% | 80,000 G |
| 駐車場 | 集客 -10% | 100,000 G |
| 景品交換所 | 単価 -10% | 20,000 G |

`shop.interior` に既存のインテリアレベルがある (`restroom`, `parking` 等) ので、
ここに `broken: boolean` フィールドを追加して壊れ状態を扱う。

### M4: 客層システム (Phase 4c)

- `shop.customerQuality`: "good" | "normal" | "bad"
- 設備状態に応じて定期的に推移
- `tickSimulation` の新規客単価と台消耗速度に乗算

### M5: CPU 覆面調査員 (Phase 5)

- CPU 店舗の顧客が プレイヤー店舗に訪問
- 評価レポートが自動生成されて `shop` に `reviews[]` として蓄積
- 評価平均で「店舗ランク」が上下
- 高ランクでガチャ無料券ボーナス

## 7. UI 上の画像サイズ選択

[MachineThumb.tsx](game/src/components/MachineThumb.tsx) は以下の解像度:

| `size` prop | 用途 |
